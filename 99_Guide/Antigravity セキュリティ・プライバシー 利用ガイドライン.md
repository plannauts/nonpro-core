本ドキュメントは、AIコーディングアシスタント「Antigravity」を安全に利用するための推奨事項、設定、およびデータの取り扱いに関するベストプラクティスをまとめたものです。

## 1. 基本的な考え方

- **「入力データは学習利用の可能性がある」という前提に立つ**
    - エンタープライズ契約などを除き、デフォルトではサービス改善や学習のためにデータが利用される可能性があります。
    - **「流出した場合に取り返しがつかない情報」は入力しない**のが最も安全なアプローチです。
- **ローカルファイルへのアクセス権限を最小化する**
    - AI自体が悪意を持たなくても、プロンプトインジェクションなどの攻撃に備え、不要なファイルへのアクセス権は与えないようにします。

## 2. 推奨される設定 (Settings)

Antigravityの設定画面 (`Settings`) において、以下の設定を確認・変更することを推奨します。

### 最優先設定

- **Account > Enable Telemetry** (テレメトリの有効化)
    - 推奨: **OFF**
    - 理由: Googleへの使用状況データの送信を直接制御する設定です。OFFにすることで、不必要なデータの外部送信を防ぎます。

### 権限設定 (現在のデフォルト OFF を維持)

以下の設定は、利便性と引き換えにセキュリティリスクを高めるため、**OFFのまま**にすることを強く推奨します。

- **Agent > Agent Non-Workspace File Access**: ワークスペース外のファイルへのアクセス。
- **Agent > Agent Gitignore Access**: `.gitignore` されたファイル（`.env` など）へのアクセス。
- **Tab > Clipboard Context**: クリップボードの中身を読み取る機能（パスワードコピー時のリスク）。

### 安全装置

- **Agent > Terminal Command Auto Execution**: ターミナルコマンドの自動実行。
    - 推奨: **Request Review** (都度確認)
    - 理由: 意図しないファイルの削除や外部との通信をブロックするため。

## 3. 入力を避けるべき情報リスト

AIへの指示（プロンプト）や読み込ませるファイルには、以下の情報を含めないように徹底してください。

### 🔴 認証情報 (Credentials)

最も危険度が高い情報です。

- APIキー、アクセストークン
- パスワード
- SSH公開鍵・秘密鍵、SSL証明書の秘密鍵
- データベースの接続文字列 (User/Passが含まれるもの)

### 🔴 個人を特定できる情報 (PII)

- 実名、住所、個人の電話番号、メールアドレス
- マイナンバー、免許証番号、パスポート番号
- クレジットカード情報、銀行口座番号
- 医療情報、病歴

### 🔴 インフラ・セキュリティ構成

攻撃者に手の内を明かすことになります。

- 社内ネットワークのIPアドレス体系、ネットワーク図
- VPN設定ファイル
- セッションID、Cookieの生データ、認証用トークン
- ファイアウォールの詳細なルール設定

### 🔴 経営・ビジネス上の機密 (Business Critical)

- 未公開の決算情報、M&A計画
- 特許出願前の発明、未発表プロダクトの詳細スペック
- NDA締結先の顧客リストや契約書
- 未修正の脆弱性レポート（具体的な攻撃パスを含むもの）

## 4. 安全に利用するためのテクニック

どうしても機密性の高いコードについて相談したい場合は、情報を **「抽象化」** または **「ダミー化」** してください。

### 方法1: 環境変数の利用

コード内に直接キーを書かず、環境変数や設定ファイルに外出しし、その設定ファイルは `.gitignore` に追加してAIから見えないようにします。

# 悪い例

api_key = "sk-12345abcdef..."

# 良い例

import os

api_key = os.getenv("API_KEY")

### 方法2: データのダミー化

実際の顧客データではなく、構造だけ同じダミーデータを作成して指示を出します。

- **実データ:** `{"user": "田中太郎", "mail": "tanaka@example.com"}`
- **ダミー:** `{"user": "test_user_01", "mail": "dummy@test.local"}`

### 方法3: ロジックの抽象化

「A社の極秘プロジェクトのデータ処理」として相談するのではなく、「このようなJSON構造を持つデータを、条件XでフィルタリングするPython関数」として一般化して相談します。
